## Acuerdos del Sprint Planning

1. **Estrategia de validacion**: dado que el equipo se redujo a 2 integrantes (Jose Amaro + Yago Caldas), se decidio enfocar la validacion del producto en el segmento principal *Cliente urbano que requiere servicios tecnicos*. El rol del Trabajador se simulara internamente para cerrar los flujos end-to-end, manteniendo todas las funcionalidades operativas en ambos perfiles.
2. **Reparto de responsabilidades**:
   - Jose Amaro: deploy publico (backend en Render, frontend en Vercel), gestion del tag `v1.0.0`, secciones 5.1, 5.2.1, 5.2.2, 5.3 del informe.
   - Yago Caldas (Scrum Master + Frontend Lead): implementacion Angular 17 + Material, pruebas unitarias JUnit, F03 Mi perfil end-to-end, capitulo 5.2.3 (Sprint 3) del informe.
3. **GitFlow estricto**: features merge a `develop`; merge a `main` solo en releases con tag formal. Conventional Commits obligatorio.
4. **Definition of Done del Sprint**:
   - Codigo mergeado a `develop` via Pull Request con al menos 1 revisor
   - Tests `BUILD SUCCESS`
   - Trazabilidad explicita entre funcionalidad/US y endpoint/componente
   - Documentado en el Sprint Backlog 3 con estado actualizado
5. **Tecnologia agregada respecto al Sprint 2**:
   - Angular 17 standalone + Angular Material 17 (indigo-pink)
   - Docker multi-stage con nginx para el frontend
   - Render Blueprint (`render.yaml`) para deploy backend
   - Vercel para deploy frontend
