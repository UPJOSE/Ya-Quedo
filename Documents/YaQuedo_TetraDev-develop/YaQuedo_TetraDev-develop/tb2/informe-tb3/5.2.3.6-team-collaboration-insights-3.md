# 5.2.3.6. Team Collaboration Insights

## Jose Emanuel Amaro Saravia

Durante el Sprint 3, la dinamica de trabajo se redujo en tamano (Austin y Diego dejaron el curso a inicios de junio) pero gano en profundidad: pudimos cubrir el alcance comprometido manteniendo el ritmo de los entregables anteriores. Mi rol se concentro en el deploy publico de la aplicacion y en la consolidacion del informe en sus secciones transversales. Para hacer viable el despliegue del backend frente a las restricciones de la cuenta personal de Yago (GitHub Actions con billing bloqueado), aproveche mi cuenta para conectar el repositorio a Render como Blueprint via `render.yaml`, lo que provisiono base de datos administrada + servicio Docker en un solo paso. Para el frontend Angular configuramos Vercel apuntando al subdirectorio `tb2/frontend/yaquedo-web/`.

El punto mas valioso de esta etapa fue la disciplina de Conventional Commits y GitFlow. Cada feature branch entro por Pull Request a `develop`, con revisor humano, y el cierre del sprint se sello con el tag formal `v1.0.0` sobre `main`. Esto deja un historial limpio, auditable, y simplifico la generacion de evidencias para el informe (secciones 5.2.3.3 y 5.2.3.5 estan trazadas directamente al `git log`).

Como mejora para el siguiente entregable, identifico que en futuros sprints conviene tener un pipeline de CI minimo (build + test en GitHub Actions) para que las PRs lleven badge de verde/rojo automatico antes del review humano. En este sprint nos limitamos a ejecutar los tests localmente porque la billing de Actions estaba bloqueada en la cuenta principal del repo, pero migrarlo al fork institucional o agregar un workflow tipo manual sigue siendo una ganancia razonable de cara al TP final.

## Ernesto Yago Caldas Zapata

Desde mi rol de Scrum Master y Frontend Lead, el Sprint 3 fue el sprint mas intenso del proyecto. Asumi la implementacion completa del frontend Angular 17 standalone con Angular Material (12 componentes, AuthInterceptor, AuthGuard, 4 modelos sincronizados con el backend), las pruebas unitarias del backend (31 tests con JUnit 5 + Mockito que cubren las 11 User Stories con escenarios exito/alternativo/error), y la redaccion del capitulo 5.2.3 del informe. Tambien ejecute la funcionalidad F03 "Gestion de perfil" end-to-end (DTOs + mappers + endpoints PATCH en backend + componente MiPerfil en frontend) que cierra los 4 items que el profesor pidio para el sprint.

Lo que mejor funciono fue trabajar en paralelo con Jose: mientras yo cerraba la integracion frontend-backend (corrigiendo el contrato real `/api/workers` `/api/requests`, alineando los modelos Angular a los `Page<>` paginados de Spring Data y los campos reales `nombres/apellidos` + `calificacionPromedio`), Jose preparaba el deploy y consolidaba las secciones que dependian del informe TB2 previo. Esto evito cuellos de botella.

Una oportunidad de mejora clara: el primer build del frontend descubrio el desfase de contrato a las 11pm de la noche del entregable, porque no habiamos hecho una verificacion end-to-end *del codigo Angular contra el backend real* antes — solo de cada uno por separado. Para el TP final voy a incluir un test de humo manual (login + listar trabajadores + crear solicitud) en el Definition of Done de cada feature branch del frontend, no solo del backend. Es 5 minutos de Postman + navegador que ahorran una hora de busqueda de bugs de modelo a ultimo momento.

Otra leccion: ahora que el equipo es de 2, los repositorios de referencia del profesor (`hampcodes/pagoya-api` + `hampcodes/pagoya-web`) demostraron ser una gran ancla. Alinear la estructura del repo (root limpio, `render.yaml` para Render, `vercel.json` para Vercel, Postman collection commited al root del backend, profiles `local` + `prod`) no solo mejoro el orden, sino que reduce el riesgo de que el profe se pierda al evaluarnos.
